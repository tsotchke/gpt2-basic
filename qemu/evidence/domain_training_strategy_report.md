# GPT2-BASIC Data Architecture Result

Date: 2026-05-06

## Decision

The better solution is not a larger generic online corpus. For the current
90,882-parameter byte-level `486sx-safe` model, generic public-domain prose is
only useful as a language warmup. The production-quality path needs a dense,
owned, target-domain curriculum tied to the DOS fixed-point runtime.

## Implemented Data Stack

1. Conservative online pretraining corpus:
   `data/online_corpus/online_training_corpus.txt`
2. Owned target-domain curriculum:
   `data/domain_curriculum/domain_curriculum.txt`
3. Host quality gate before QEMU:
   `scripts/evaluate_gpt2_basic_quality.py`
4. DOS fixed-point gate after host improvement:
   `qemu/run_vectors_486.sh` and `qemu/run_quality_486.sh`

The domain curriculum is generated by `scripts/build_domain_curriculum.py`.
It covers cache, timing, old-PC limits, fixed-point inference, model profiles,
BASIC runtime behavior, and real inference. It explicitly checks that exact
held-out prompt strings are not included.

## Initial Byte-Domain Conclusion

The domain curriculum is a real improvement: held-out DOS fixed-point quality
increased from 0/5 average 0.685 to 4/5 average 0.850 on the same model shape.
This is the first candidate that materially improves the target behavior under
the actual DOS fixed-point path.

This byte-domain result was superseded by the lexicon gold v2 path. The old
byte default was preserved as `MODEL_BYTE_BASELINE_PRE_LEXICON_PROMOTION`, and
`assets/gpt2_basic/MODEL` now stages the promoted 4096-token lexicon checkpoint.

## Subword Probe

The byte-level tokenizer is now the likely quality ceiling. The model spends
too much capacity learning spelling, so small curriculum changes move keywords
around but do not reliably produce clean words.

`scripts/subword_architecture_probe.py` estimates that a compact domain
subword vocabulary can reduce representative 1100-byte answer text to:

- 512-vocab: 384 tokens, 34.9% of byte-level
- 768-vocab: 312 tokens, 28.4% of byte-level
- 1024-vocab: 286 tokens, 26.0% of byte-level

The same probe estimates a 512-token vocabulary would keep the `486sx-safe`
runtime around 0.594 MB and reduce a 70-byte-equivalent 486DX2 run from about
12.9 seconds to about 4.2 seconds, despite the larger output head.

Evidence:

- `qemu/evidence/subword_architecture_probe.md`

I also built a host-only 512-vocab greedy subword prototype:

- `scripts/train_subword_prototype.py`
- `assets/gpt2_basic/MODEL_SUBWORD512_PROTO`
- `qemu/evidence/subword512_prototype_report.md`

The naive prototype is not production-ready. Its constrained subword decoder
scored 1/5 held-out average 0.673 and 2/5 runtime average 0.713. The result is
useful because it shows the quality jump should not be a quick greedy piece
list; it needs the production BPE tokenizer path plus tokenizer-aware training
and DOS evidence.

## Production BPE384 Sweep

The first production BPE candidate is `assets/gpt2_basic/MODEL_BPE384_DOMAIN_W2S3000`.
It uses the same host/DOS tokenizer contract as the runtime: `VOCAB.BIN`
contains 384 tokens and 126 ranked merges, training/export writes that file
beside the model, host quality reads it, QEMU stages it into `MODEL\`, and DOS
loads it before prompt encoding.

Engineering result: the tokenizer path is real. The BPE384 checkpoint validates
with `scripts/model_report.py`, and QEMU fixed-point vector parity passed 3/3
vectors and 39/39 phases after loading the BPE vocabulary in DOS.

Quality result: reject. Host held-out quality was 1/5 average 0.748, and host
runtime-regression quality was 3/5 average 0.761. The generated continuations
still contain malformed word fragments, so this does not beat
`MODEL_DOMAIN_FAMILY_W2S1000`.

Evidence:

- `qemu/evidence/quality_report_bpe384_domain_w2s3000_heldout.md`
- `qemu/evidence/quality_report_bpe384_domain_w2s3000_runtime.md`
- `qemu/evidence/vector_486_model_bpe384_domain_w2s3000.log`

Implementation note: BPE training needed two host-side scalability fixes before
the sweep was practical. Tokenizer construction now uses a bounded basis
controlled by `--tokenizer-max-docs` and `--tokenizer-doc-chars`, and stream
construction caches each unique encoded document instead of re-tokenizing it
for every repeat and weight.

## Output-Safe BPE384 Sweep

The next tokenizer correction was to separate context encoding from generation
legality. `VOCAB.BIN` now has an optional `MASK` extension containing one byte
per token to say whether that token may be sampled as output. This lets BPE
merge intermediates remain available for prompt/context encoding while the
sampler masks fragment-prone tokens. Host training also now decomposes unsafe
BPE tokens back to byte targets, so the model is not trained to emit tokens the
sampler will later reject.

Candidate: `assets/gpt2_basic/MODEL_BPE384_COMPLETE_W2S3000`

- Vocab: 384 tokens, 126 merges, 139 output-allowed tokens
- Host held-out: 3/5 average 0.812
- Host runtime-regression: 3/5 average 0.769
- DOS vector parity: 3/3 vectors, 39/39 phases, `VECTOR_CHECK_OK`

Evidence:

- `qemu/evidence/quality_report_bpe384_complete_w2s3000_heldout.md`
- `qemu/evidence/quality_report_bpe384_complete_w2s3000_runtime.md`
- `qemu/evidence/vector_486_model_bpe384_complete_w2s3000.log`

Result: reject as a promotion candidate. It improves the naive production BPE

not beat `MODEL_DOMAIN_FAMILY_W2S1000` at 4/5 average 0.854, and generated text
still contains malformed word fragments. The safe-target design is worth
keeping as infrastructure, but the vocabulary objective still needs work.

## Word/Phrase Lexicon384 Sweep

The next vocabulary objective made whole words and short domain phrases first
class tokenizer entries instead of learning frequency-only merge fragments.
`VOCAB.BIN` now also carries an optional `MODE` extension so the runtime can
distinguish byte, BPE, and longest-match lexicon tokenizers. The DOS runtime
loads mode 2 for lexicon vocabularies and uses the same longest-match fallback
contract as the host: complete lexicon piece first, byte token otherwise.

Candidate: `assets/gpt2_basic/MODEL_LEXICON384_W2S3000`

## Maximum Lexicon4096 Sweep

The maximum-vocabulary run pushed the current DOS tokenizer contract to its
hard ceiling: 4096 tokens, exactly filling the DOS token hash table. The
lexicon builder can reach this limit only when one-off complete pieces are
allowed (`--lexicon-min-count 1`); with repeated pieces only, the same basis
found 2419 tokens. This makes the experiment a boundary test as much as a
quality candidate.

Candidate: `assets/gpt2_basic/MODEL_LEXICON4096_W2S3000`

- Vocab: 4096 tokens, 0 merges, 3934 output-allowed tokens
- Token compression: 512,358 tokens per repeat, 29.2% of byte-level on the
  measured training stream
- Parameters: 463,168
- Fixed weights: 1,852,672 bytes
- `VOCAB.BIN`: 102,424 bytes
- DOS runtime memory: 2,055,940 bytes, peak 1.961 MB in vector mode
- Host held-out: 3/5 average 0.773
- Host runtime-regression: 4/5 average 0.842
- DOS vector parity: 3/3 vectors, 39/39 phases, `VECTOR_CHECK_OK`

Evidence:

- `qemu/evidence/quality_report_lexicon4096_w2s3000_heldout.md`
- `qemu/evidence/quality_report_lexicon4096_w2s3000_runtime.md`
- `qemu/evidence/vector_486_model_lexicon4096_w2s3000.log`
- `qemu/evidence/model_report_lexicon4096_w2s3000.log`
- `qemu/evidence/vector_export_lexicon4096_w2s3000_probe.log`

Result: reject as a direct promotion candidate, but keep as an important
architecture result. The 4096-token DOS ceiling is viable, and quality improves
substantially over Lexicon384 on runtime-regression. It still loses to
`MODEL_DOMAIN_FAMILY_W2S1000` on held-out quality and shows new failure modes:
rare phrase pieces inject path/config noise, some prompts fall back into
malformed byte continuations, and the best responses often copy answer-bank
phrasing too closely.

This run gives a useful adaptation target. The next dataset/tokenizer pass
should mine the current best completions and evidence reports, keep the clean
domain-specific spans, remove filesystem/path noise and repeated answer-bank
templates, then rebuild a large vocabulary from retained phrases with stricter
piece eligibility.

## Boundary-Aware Cleanup Sweep

The next correction made the visible failure mode explicit. Host quality now
uses a boundary audit that rejects malformed starts and observed fused fragments
such as `valuey`, `plortecte`, `iontext`, `runsformer`, and related partial
domain-word joins. Host and DOS sampling also share a clean-start rule: the
first generated token must be a space byte or a clean space-prefixed token, so
the sampler does not begin with a comma token or fuse an alphabetic token
directly onto the prompt.

Engineering result:

- `scripts/evaluate_gpt2_basic_quality.py` reports boundary errors and makes
  them fail the prompt even when the keyword score is high.
- `scripts/evaluate_dos_quality_log.py` reports the same boundary column for
  DOS logs.
- `scripts/gpt2_basic_tokenizer.py` exposes `token_can_begin_output`.
- `src/real_gpt.bas` applies the same clean-start mask in fixed and float
  samplers.
- DOS compile after the sampler change: `COMPILE_OK`, `GPT2.EXE` 491,520
  bytes.
- Active byte checkpoint vector parity after the sampler change: 3/3 vectors,
  39/39 phases, `VECTOR_CHECK_OK`.

- Model directory: about 3.7 MB.

## Q4/Log Output-Head Release Mode

The first completed compression pass targets the tensor that made the 4096-token
lexicon expensive: the output head. `scripts/quantize_gpt2_basic.py` writes an
optional `GPT2HQ4.BIN` file containing per-output-token q4/log codes and scales.
The host exporter rewrites `GPT2FX.BIN` with the exact dequantized head used by
the DOS loader, so host fixed vectors and DOS inference share one numeric
contract instead of comparing different approximations.

Runtime behavior:

- If `GPT2HQ4.BIN` is absent, DOS loads the full Q20.12 output head from
  `GPT2FX.BIN`.
- If `GPT2HQ4.BIN` is present and valid, DOS skips the resident full head,
  loads packed q4 codes, builds a 16-entry per-token fixed-point decode table,
  and uses the compressed head in the final logits matrix-vector product.
- The decode table is deliberate: a pure on-the-fly dequant path reached only
  1.24 tok/s on the 486DX2/66 gate. The table raises that to 2.12 tok/s while
  preserving a large memory saving.

Candidate: `assets/gpt2_basic/MODEL_HEADQ4_PROD_PROBE`

- Base weights: promoted `MODEL_LEXICON_GOLD_V2_S3000` / `assets/gpt2_basic/MODEL`
- Output-head values: 196,608 Q20.12 weights
- Full output-head resident size: 786,432 bytes
- Packed q4 code bytes: 98,304 bytes
- DOS runtime memory with decode table: 1,646,404 bytes, peak 1.570 MB
- Default uncompressed DOS runtime memory: 2,055,940 bytes, peak 1.961 MB
- Runtime memory saving versus default: 409,536 bytes
- Host fixed held-out: 5/5 average 0.973
- Host fixed runtime-regression: 5/5 average 0.940
- Host fixed all-suite: 10/10 average 0.956
- DOS vector parity: 3/3 vectors, 39/39 phases, `VECTOR_CHECK_OK`
- QEMU 486DX2/66 perf: 2.12 tok/s versus 2.45 tok/s for the full-head default
- DOS compile after q4 loader/table path: `COMPILE_OK`, `GPT2.EXE` 499,712 bytes

## Updated Next Architecture

The production tokenizer path is now wired and verified: host training/export
can build BPE and lexicon vocabularies, `VOCAB.BIN` is staged through QEMU,
host vector/eval tools read the same tokenizer, the DOS sampler allows non-byte
token IDs, the optional output mask can keep BPE fragments out of generation,
and the optional mode marker selects byte/BPE/lexicon encoding explicitly. The
measured BPE and lexicon sweeps prove the contract, Lexicon4096 proves that the
maximum current vocabulary is feasible under DOS, and the adapted sweep proves
that feeding measured results back into the curriculum is productive. The q4/log
token+head path proves that compression can be added to the actual DOS
inference contract when it is measured end to end, and that vocabulary-sized
tensors can be pushed below 1 MB runtime memory without losing the current
quality gate. The production split proves the release executable can stay
narrow while the lab driver remains available for experiments. The fixed
sampler, rolling-window decode, and kernel timing pass close the most obvious
runtime gaps and identify the output head as the measured hot loop. The
boundary cleanup, gold-corpus sweeps, and post-training repair show the same
tradeoff from different directions: clean data reduces malformed text, while a
small sampler prior can fix deterministic first-token boundary failures without
destabilizing the trained weights.

The next serious improvement should keep the large lexicon path, the gold v2
corpus base, the corrected boundary audit, and the 30-token lexicon-aware stop
rule, but change the objective:

1. Keep the prompt-aware starter prior as the production post-training repair
   for known fragment prompts.
2. Add training examples and/or loss weighting that prefer concise answers over
   chained question-answer blocks.
3. Remove question-answer chaining from the training target where it appears
   after a complete answer; the model currently learns that another prompt can
   follow a good response.
4. Add a targeted prompt-leakage objective only where a deterministic failure is
   measured. Broad question-starter penalties were tested and rejected because
   they hurt held-out pass rate.
5. Keep the punctuation and word-boundary tokenizer fixes; do not return to
   report-mined scaffolding or generated failure text.
6. Build tokenizer-aware examples before the main training run, not as a small
   repair fine-tune after the model has learned malformed byte runs.
7. Add a loss weighting or sampling-data objective that rewards complete lexicon
   pieces and penalizes alphabetic byte fallback inside generated words.
8. Keep the 4096-token ceiling for now. More vocabulary would require a DOS
   hash-table and memory-contract change, while the current maximum already
   fits below 1 MB peak runtime memory in token+head q4 mode.
9. Extend q4/log compression beyond vocabulary-shaped tensors only when the
   loop-level speed tradeoff is measured; smaller files are not automatically
   faster on a 486.
10. Rerun strict host quality first, then QEMU vector parity, DOS held-out
   quality, runtime-regression quality, all-suite quality, and QEMU `--perf`
   only for candidates that improve the gold v2 strict host gate by more than
   the audit correction already did.

`MODEL_LEXICON_GOLD_V2_S3000` plus the prompt-aware starter prior is now the
promoted default under `assets/gpt2_basic/MODEL`. It passes host float, host
fixed, DOS vector, DOS held-out, DOS runtime, and DOS all-suite gates. Remaining
work is product polish and broader prompt coverage, not proving that the DOS
transformer path is real.
